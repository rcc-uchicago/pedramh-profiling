# PanguWeather repo cleanup — instructions

**For:** the maintainer of `git@github.com:envfluids/PanguWeather.git` (jesswan)
**From:** the Polaris/PBS bring-up (see `polaris_pbs_notes.md` in `rcc-uchicago/pedramh-profiling`)
**Date:** 2026-07-14

`PanguWeather` tracks **68.7 MB of scheduler job logs across 102 files** — regenerated
on every run and never useful in git. This doc is the exact, verified recipe to stop
tracking them.

Everything below was measured against the real repo, not guessed. **Read §1 first — one
category of large files looks like junk but is required.**

---

## 1. ⚠️ DO NOT remove `constant_mask/*.npy` — the model needs them

There is **55.9 MB of `.npy`** in the repo. It is **NOT junk** — these are static boundary
fields loaded at runtime:

| file | size | actually loaded by (`np.load`) |
|---|---|---|
| `soil_type.npy` | 7.3 MB ×3 | `v1.0/train.py:126-128`, `v1.0/train_profiling.py`, `v1.0/train_precip.py`, `v2.0/train.py`, `v2.0/long_inference.py`, `Pangu-UC/examples/pangu_plasim/data_utils.py:153` |
| `land_mask.npy` | 7.3 MB ×3 | same |
| `topography.npy` | 4.0 MB ×3 | same |

(The path comes from the config's `mask_dir`, e.g. `v1.0/config/PANGU.yaml:26` →
`data/constant_mask`. Note `v1.0/networks/pangu_lite.py` only *mentions* the masks in a
comment on line 20 — it receives them as tensors and never reads the files; an earlier
draft of this table miscredited it.)

A blanket `*.npy` ignore rule **would break the model.** (For reference, `pedramh-profiling`
hits the same issue and solves it with an explicit carve-out:
`!s2s-lightning/data/constant_mask/*.npy`.)

> **Optional, separate cleanup:** the same 18.6 MB is **triplicated** across
> `data/constant_mask/`, `Pangu-UC/examples/pangu_plasim/data/constant_mask/`, and
> `Pangu-UC/examples/pangu_lite/data/constant_mask/` — ~37 MB reclaimable by deduping to
> one copy + symlinks/config. That is a code refactor, not a gitignore change. Don't
> combine it with the log cleanup below.

## 2. The actual junk: scheduler job logs (68.7 MB, 102 files)

The repo mixes **two** naming conventions, which is why a naive rule misses half of them:

| convention | example | job id |
|---|---|---|
| PBS (Derecho/Polaris) | `e3sm_sfno_train.o5808412` | **after** `.o` |
| SLURM (Stampede) | `e3sm_train_3166315.e` | **before**, bare `.e` |

Biggest offenders:

## 3. Step 1 — add to `.gitignore`

Append to `PanguWeather/.gitignore`:

```gitignore
# --- Scheduler job logs (regenerated every run; never useful in git) ---
# PBS (Derecho/Polaris):  <name>.o<jobid> / <name>.e<jobid>
*.o[0-9]*
*.e[0-9]*
# SLURM (Stampede):       <name>_<jobid>.o / <name>_<jobid>.e
# NOTE: the char before .o/.e must be a DIGIT, so a C object file (foo.o) is NOT matched.
*[0-9].o
*[0-9].e
*.out
*.err
outs/

# --- Crash artifacts ---
core.*

# --- Run outputs / generated data ---
results/
checkpoints/
wandb/
events/
*.nsys-rep
*.sqlite
# Generated E3SM aux data (regenerated by v2.0/polaris_prepare_e3sm_stats.py)
v2.0/polaris_data/
*.nc

# NOTE: do NOT add "*.npy" — see section 1. constant_mask/*.npy are required inputs.
```

**Verified:** these four log patterns match **102/102** job logs with **zero** false
positives on source or data files.

## 4. Step 2 — stop tracking them

`.gitignore` only affects *untracked* files, so the already-committed logs must be removed
from the index. **Dry-run first:**

```bash
cd PanguWeather
git ls-files | grep -E '\.o[0-9]+$|\.e[0-9]+$|[0-9]\.o$|[0-9]\.e$' | tee /tmp/pangu_logs.txt | wc -l
# expect: 102     <-- review /tmp/pangu_logs.txt before continuing
```

Then remove from the index (files stay on your disk — `--cached`):

```bash
git rm --cached --quiet $(cat /tmp/pangu_logs.txt)
git add .gitignore
git commit -m "Stop tracking scheduler job logs (68.7 MB, 102 files)

They are regenerated on every run. Covers both conventions:
PBS <name>.o<jobid> and SLURM <name>_<jobid>.o.
Does NOT touch constant_mask/*.npy, which are required runtime inputs."
git push
```

## 5. Step 3 — verify

```bash
git ls-files | grep -cE '\.o[0-9]+$|\.e[0-9]+$|[0-9]\.o$|[0-9]\.e$'   # expect: 0
git ls-files | grep -c 'constant_mask.*\.npy'                          # expect: 9  (still tracked!)
git status --short | head                                              # logs now ignored, not "??"
```

## 6. ⚠️ What this does NOT do — it will not shrink the clone

**Steps 3–4 stop the logs being *checked out*, but the blobs stay in history forever, so
`git clone` downloads them regardless.**

Proof from this repo: pushing it triggered

```
remote: warning: File v2.0/outs/anvil_1.out is 53.15 MB; this is larger than
GitHub's recommended maximum file size of 50.00 MB
```

…and **`v2.0/outs/anvil_1.out` is not in the current tree at all** — it was deleted long
ago, yet still ships with every clone.

To actually reclaim the space you must **rewrite history**:

```bash
pip install git-filter-repo
git filter-repo --invert-paths \
  --path-glob '*.o[0-9]*' --path-glob '*.e[0-9]*' \
  --path-glob '*[0-9].o'  --path-glob '*[0-9].e' \
  --path-glob 'v2.0/outs/*'
git push --force
```

**Read this before running it:**
- It **rewrites every commit SHA**. Everyone must **delete and re-clone**; stale clones
  cannot pull.
- **Open PRs and issue links to commits break.**
- Anyone who forked keeps the old objects.
- `--force` push to a shared branch is destructive — coordinate first.

**Recommendation:** do §3–§4 now (stops the bleeding, keeps SHAs stable). Only do §6 if
clone size becomes a real complaint — and if so, do it **before** more forks accumulate.

**Two `train.py` gotchas worth knowing regardless of Polaris:**
1. **`--debug` is single-GPU only** — it hardcodes `params['world_size'] = 1`, so under
   `torchrun --nproc_per_node=4` every rank initialises as rank-0-on-GPU-0 and they all OOM
   the same card. Bound a short run with **`--epochs 1`** instead.
2. On a Lustre filesystem, writing the climatology needs **`HDF5_USE_FILE_LOCKING=FALSE`**
   or it dies with `BlockingIOError: unable to lock file`.
